# TS Proxy — Implementation Plan

## Overview

Criar um micro-servidor Node.js standalone que atua como relay com auto-reconexão para streams IPTV que entregam segmentos TS individuais (~10s) via CDN com `Connection: close`. O proxy recebe a URL original encodada, segue todos os redirects, pipa os bytes para o cliente, e reconecta automaticamente ao servidor de origem quando a conexão fecha — entregando ao player (VLC/ExoPlayer) um stream TS aparentemente contínuo.

## Motivation & Context

Servidores IPTV do tipo `poupadefrutas.shop` (Xtream Codes sem HLS) entregam streams através de uma cadeia de redirects que culmina em um único segmento TS de ~10s num CDN com URL assinada e `Connection: close` explícito. Cada request à URL de origem gera um novo token CDN. VLC e ExoPlayer tratam o `Connection: close` como fim de stream e param a reprodução. O proxy resolve isso mantendo a conexão com o cliente aberta e reconectando silenciosamente no servidor de origem a cada segmento.

Problema confirmado experimentalmente:
- `http://site.poupadefrutas.shop:80/user/pass/62169` → 3 redirects → CDN `.ts` com `Connection: close` → para em ~10s
- `.m3u8` → 405 Not Allowed (servidor não implementa HLS)
- VLC e Android TV (ExoPlayer) ambos afetados

## Current State Analysis

Projeto novo — não existe nada ainda. Referências ao comportamento observado:

- Redirect chain confirmado via `curl -vL`: 3 hops, último é `Connection: close`
- Redirect 3 da cadeia é **relativo** (`/user/pass/id?token=...`) — requer resolução relativa ao host atual
- Token CDN é time-limited — o proxy deve reconectar sempre à URL **original** (não à URL CDN), para obter novo token a cada segmento
- Latência esperada entre segmentos: ~200ms de pausa entre reconexões é suficiente

#### 3. `server.js`

Lógica completa em ~70 linhas usando apenas módulos built-in (`http`, `https`, `url`):

```javascript
'use strict';

const http = require('http');
const https = require('https');
const { URL } = require('url');

const PORT = process.env.PORT || 3000;
const MAX_REDIRECTS = 10;

/**
 * Faz GET em targetUrl, segue redirects manualmente (incluindo relativos),
 * e pipa o body para `res`. Resolve quando o upstream fecha a conexão.
 */
function fetchAndPipe(targetUrl, res, redirectCount = 0) {
    return new Promise((resolve) => {
        if (redirectCount > MAX_REDIRECTS) {
            console.error('[proxy] Too many redirects for', targetUrl);
            return resolve();
        }

        let parsed;
        try { parsed = new URL(targetUrl); }
        catch (e) { console.error('[proxy] Invalid URL:', targetUrl); return resolve(); }

        const client = parsed.protocol === 'https:' ? https : http;

        const req = client.get(targetUrl, {
            headers: { 'User-Agent': 'Mozilla/5.0', 'Connection': 'close' },
            timeout: 20000,
        }, (upstream) => {
            // Seguir redirects (incluindo relativos)
            if (upstream.statusCode >= 300 && upstream.statusCode < 400 && upstream.headers.location) {
                upstream.resume(); // descartar body
                const location = upstream.headers.location;
                const next = location.startsWith('http')
                    ? location
                    : `${parsed.protocol}//${parsed.host}${location}`;
                fetchAndPipe(next, res, redirectCount + 1).then(resolve);
                return;
            }

            if (upstream.statusCode !== 200) {
                console.error('[proxy] Upstream error:', upstream.statusCode, targetUrl);
                upstream.resume();
                return resolve();
            }

            upstream.on('data', (chunk) => {
                if (!res.writableEnded) res.write(chunk);
            });
            upstream.on('end', resolve);
            upstream.on('error', resolve);
        });

        req.on('error', resolve);
        req.on('timeout', () => { req.destroy(); resolve(); });
    });
}

const server = http.createServer(async (req, res) => {
    // Health check
    if (req.url === '/health') {
        res.writeHead(200);
        res.end('ok');
        return;
    }

    const match = req.url.match(/^\/ts\/([A-Za-z0-9_-]+=*)/);
    if (!match) {
        res.writeHead(400);
        res.end('Usage: GET /ts/{base64url-encoded-url}');
        return;
    }

    let targetUrl;
    try {
        targetUrl = Buffer.from(match[1], 'base64url').toString('utf8');
        new URL(targetUrl); // valida
    } catch {
        res.writeHead(400);
        res.end('Invalid encoded URL');
        return;
    }

    console.log('[proxy] Stream start:', targetUrl);

    res.writeHead(200, {
        'Content-Type': 'video/mp2t',
        'Transfer-Encoding': 'chunked',
        'Connection': 'keep-alive',
        'Access-Control-Allow-Origin': '*',
        'Cache-Control': 'no-cache',
    });

    req.on('close', () => {
        console.log('[proxy] Client disconnected:', targetUrl);
        res.end();
    });

    // Loop de reconexão: sempre volta à URL original para obter novo token CDN
    while (!res.writableEnded) {
        await fetchAndPipe(targetUrl, res);
        if (!res.writableEnded) {
            // Pausa breve entre segmentos para evitar hammering
            await new Promise(r => setTimeout(r, 300));
        }
    }

    console.log('[proxy] Stream end:', targetUrl);
});

server.listen(PORT, () => console.log(`[ts-proxy] listening on :${PORT}`));
```

## Phase 2: Deployment Configs

### Overview

Criar `Dockerfile`, `fly.toml`, `railway.toml` e `README.md`. Ao final desta fase o projeto está pronto para `fly deploy` e `railway up` sem nenhuma configuração adicional.

### Motivation for this phase

Configurações de deploy são separadas da lógica porque requerem decisões específicas de plataforma (região, memória, scaling) que podem mudar sem alterar o servidor em si.

### Changes Required

#### 1. `Dockerfile`

```dockerfile
FROM node:18-alpine
WORKDIR /app
COPY package.json ./
COPY server.js ./
EXPOSE 3000
CMD ["node", "server.js"]
```

Sem `npm install` — zero dependências externas.

#### 2. `fly.toml`

```toml
app = "ts-proxy"          # substituir pelo nome escolhido no fly launch
primary_region = "gru"    # São Paulo — mais próximo do usuário BR

[build]

[http_service]
  internal_port = 3000
  force_https = true
  auto_stop_machines = "stop"
  auto_start_machines = true
  min_machines_running = 0

  [http_service.concurrency]
    type = "connections"
    hard_limit = 25
    soft_limit = 20

[[vm]]
  size = "shared-cpu-1x"
  memory = "256mb"
```

**Nota sobre `auto_stop_machines = "stop"`:** a máquina hiberna quando sem requests. Para streams longas (>5min), considerar `min_machines_running = 1` para evitar cold start durante reprodução.

#### 3. `railway.toml`

```toml
[deploy]
  startCommand = "node server.js"
  healthcheckPath = "/health"
  healthcheckTimeout = 10
  restartPolicyType = "ON_FAILURE"
  restartPolicyMaxRetries = 3
```

Railway detecta Node.js automaticamente via `package.json`, sem necessidade de Dockerfile.

## Testing Strategy

### Manual Testing Steps

1. Encodar URL problemática: `node -e "console.log(Buffer.from('http://site.poupadefrutas.shop:80/q2Y7tqVAUv/dnuavyuZV5/62169').toString('base64url'))"`
2. Iniciar proxy local: `node server.js`
3. Abrir no VLC: `http://localhost:3000/ts/{encoded}` — deve reproduzir por mais de 30s sem parar
4. Observar logs: devem mostrar reconexões a cada ~10s
5. Fechar VLC: log de `Client disconnected` deve aparecer imediatamente

### Edge Cases to Verify

| Caso | Como testar | Esperado |
|------|-------------|----------|
| URL inválida | `GET /ts/nao-e-base64url` | 400 Bad Request |
| Servidor upstream down | URL que não responde | Loop tenta, cliente recebe silêncio, não crasha |
| Cliente desconecta no meio | Fechar VLC mid-stream | `res.writableEnded = true` → loop para |
| Redirect relativo | URL do poupadefrutas (tem redirect relativo) | Segue corretamente sem erro |
| Muitos redirects | Loop de redirect | Para após `MAX_REDIRECTS = 10` |

## Performance Considerations

- **Memória por stream:** ~256KB de buffer por conexão upstream (Node streams default) — para uso pessoal com 1-3 streams simultâneos, `256mb` no Fly é suficiente
- **Bandwidth:** cada stream consome ~2-8 Mbps dependendo da qualidade do canal. Fly.io free trial: sem limite explícito de bandwidth. Railway free: 100GB/mês
- **CPU:** negligível — é puro pipe de bytes, zero processamento
